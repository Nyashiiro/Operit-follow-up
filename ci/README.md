### CI 脚本

`script/` 保存本机和外部 CI 共用的单一职责入口。后续步骤统一由 pixi 提供依赖和任务编排，平台适配层只调用这些入口。


## CI的开发规范

这些要求适用于各CI子模块的开发、构建、测试和维护命令。
目标是保证命令安全、可预期、可自动化、可观测，并能组合到未来的统一 Pixi Workspace 入口

### 文档

为了保证脚本的可维护性和可组合性

- 每个模块必须有 `<模块目录>/README.md`，说明用途、意图、输入、输出、错误处理和副作用
- 复杂或独立使用的子命令应提供 `<子命令目录>/docs/tldr_<command>.md` 和 `<子命令目录>/docs/man_<command>.md`。
- 产生结构化 JSON 输出的命令应提供 `<子命令目录>/docs/schema_<command>.json`，并声明 schema 版本与兼容性策略
- 帮助文档可使用 Markdown 渲染，统一依赖 `glow`；必须命令使用**Bold**,非必须命令使用*斜体*，同时用文本语义表达
- 脚本应有合适的注释来帮助理解其意图，对于复杂的语法糖依赖，需要注明该对象、方法所在的手册、文档/原源码位置
	- 不要把依赖文档下载到本地，除非它本身就包含在依赖包里
	- 不要引用不锁定大版本号的文档仓库

### 全局重要参数

这些内容绝大多数子模块需要使用，且必须专门维护

#### 交互模式与非交互模式

命令根据`--no-interactive`参数分为人机交互两界
- 请不要假设交互模式与非交互模式有一样的行为
- 但交互模式和非交互模式的同样命令组应当代表同一个意图
- 除非你知道你在做什么，否则不要消费掉此参数！！！

传入`--no-interactive`时
- 命令须假设自身在全自动的ci环境、脚本、自动化应用中运行，但行为和调用方类型无关
- 禁止等待用户输入、交互式选择、确认弹窗及其他隐式控制流程，所有参数需要一次显式传入一旦缺失或错误即快速失败\
- 当然非常建议您像这样使用`\`续行，这会使您的脚本更清晰，大部分编程语言可以换行编写单元操作或使用续行符
- 因为语法错误（如 缺少必要参数、参数无效、前置条件不满足或资源不足时，必须指出修复方式，指出出错的子命令以便于调查文档
- 不提供非交互模式下的配置文件，但您依然应该注意项目本身的配置文件和锁文件
- 搬运命令行本身，即完整命令因此包含完整的意图，但你依然需要完整的项目文件来复现行为
- 因此，可以使用环境变量承载敏感值，但不要用于那些包含意图，作用域和范围的量
- 建议把动作指示和进度表示规范化为一定心跳，便于包装为gui

#### 关于`--dry-run`

`--dry-run` 的语义是演习，不是一次性输出内部详细日志，也不是干一遍然后回滚
- 所有子命令应按情况维护`--dry-run`, 注意交互模式和非交互模式的行为，`--dry-run`也可以报错或停止！！！
	- 会产生对用户数据、工作区状态、构建缓存、系统环境、网络资源或外部服务产生的可观察变更的命令必须支持
- 交互模式执行`--dry-run`时依然需要与用户交互式操作，此时提示、输入标识、进度条和活动指示需要使用橙色`[dry-run]`标记
	- 本来的耗时操作，交互模式下演习时，也需要模拟阻塞和加载；阻塞时间建议常量化，不硬编码也不做成该子命令参数
- `dry-run`严格来说是个*副作用*操作，只是一般不持久化数据，但它并不能保证所有行为和真实行为一致，使用它能更了解命令，但不能保证安全和正确，切记
	- 它的真实边界是，执行必要的前置检查和有限探测，展示计划执行的操作，但不提交持久化变更，也不启动高资源消耗的实际任务
	- 写入配置、删除缓存、发布版本、上传文件、完整依赖下载、实际编译不会执行，但它依然执行真实读取本地文件和从网络下载元数据文件等行为
- 建议模块开发dry-run.py等专用配置入口，便于调整行为和观感，但仍须复用主要代码的大部分路径


### 交互需要人性化

> 人是多么伟大，又多么脆弱；多么崇高，又多么卑劣！
> 人性化是用于弥补、掩盖、满足人的缺点的特质
>

为了帮助用户快速上手命令
- 未提供子动作，或者必要参数遗失的时候，展示最后一个解析到的命令的help，并以非0状态码退出
	- 如，当你执行 pixi run debug {错误分类} ，提供错误描述：`不存在的测试类型（the <xxx> test  not exist)` 与debug的--help；执行pixi run debug github-login {提供了错误的参数}，那么 需要提供login 的 --help。
	- 我们不能假设无自动作代表all/auto，如果你使用的工具或者脚本有这种行为，可以用pixi重新包装一下
- 命令、子动作参数名称和缩写尽可能不产生歧义，help中参数按执行顺序分组，并在每组内按重要性排序；应提供合理的缩短形式以便于快速操作
- 需要在执行前先显式打印 now you're exec `<去除缩写的长形式命令>` ， 因为用户依然可能记混缩写
	- 需要按照参数语义分组重新排序，得到一个语义明确的，只改变其中可变位置参数顺序应该排序结果完全相同的长形式命令
	- 这一步需要隐去标记为敏感信息的命令，但我们依然不建议你在命令中直接通过参数传递它们
	- python库在交互式命令行本身提供隐私的命令输入行为

为了防止用户像使用魔法一样执行意外操作
- 在每个耗时和危险步骤之前，输出将执行的操作摘要；
	- 为了避免刷屏，可以直接在摘要项前刷新完成指示、动作指示、未执行指示，或在摘要项后刷新进度条
	- 分类提醒其对本地文件、构建产物、缓存、远端资源或外部状态的改变
- 显性化隐性行为，优先让用户确认
	- 准则：[隐性行为是什么](Explicitation.md)
	- 会产生对用户数据、工作区状态、构建缓存、系统环境、网络资源或外部服务产生的可观察变更的命令必须支持
	- 在可行的情况下，阻塞性提示需要在用户操作之后的最早阶段完成，这可以通过事务性预检来达到
	- 任务恢复、回滚提示需要在执行流完成，即所有可行任务都完成后触发（如移动文件创建软链接之后
- 危险行为（含资源不足和前置条件不足等）不得使用 y/N 提示（假如你一直在剪礼品卡，你大概率会剪掉一张银行卡）
	- 使用操作语义/目的作为提示，如`DELET`, `ROLLBACK`, `CANCEL`.不要强制大小写
	- 批量操作使用`ALL-`前缀，需要等待，起步五秒，每个文件加一秒，封顶60秒
	- 可以做一个通过键鼠框选、勾选的tui
	- 应当分类展示操作对象，存在多类时不应该平铺或捆绑在一个all下方
	- 为避免用户使用 y | 等方式导致执行流卡死，一旦短时间内被灌入过多（例如25次）无关项，就以恢复当前操作前原来状态为原则，即撤销当前操作元，并回传被撤销状态
	- 应该原子化的一批提示不得分开让用户选择，无关的几个提示不得捆绑选择

预防卡死，以及预防用户觉得卡死了
- 交互模式不得无状态阻塞：可估算进度时使用进度条，否则显示持续活动指示(如旋转指示)
- 应在命令文档中声明内存、磁盘等主要资源消耗、可能写入的位置和预计耗时来源
- 运行前检查当前系统、所需权限、磁盘空间和可用内存，并在交互模式明显展示
- 磁盘最低要求采用明确公式：`free >= max(total * 5%, 10 GiB)`；不同任务可提高该要求
- 所有构建任务的线程数和 JVM 堆大小必须由模块从当前环境可用资源推导，不得在 Gradle 或 Pixi 配置中写死（当前未测试，所以改为显式传入和交互式确认）
- 依赖实机资源的并行性，垃圾回收策略，内存分配策略配置等需要.gitignore
- 不合理的显式资源配置应被拒绝；仅 `--ignore-resource-limit` 可绕过，并必须醒目警告

### 集成需要规范化

#### 命令接口

- 命令必须具有清晰的 `description`
- 命令名称、子动作名称与缩写不得产生歧义
- 输出的命令摘要须隐藏密钥、令牌、密码等敏感信息
	- 使用统一的日志格式化与脱敏工具，不要自己造轮子
- JSON 模式下，JSON 仅写入 stdout；日志、提示和进度写入 stderr。
- 需要使用标准的错误码表预分类错误，建议使用类rust标准
	- 约定: 退出时你的stderr的最后一行是一个特殊的json，用来表示错误信息
- 非交互模式不得无限等待，必须提供超时参数


#### pixi规范化

目标是便于组装成统一的pixiworkspace（monorepo），我们会编写menu化tui

- 仓库根 `pixi.toml` 是统一调用入口，只指向 `ci/pixi.toml`。实际 Workspace、环境、任务和锁文件分别由 `ci/pixi.toml` 与 `ci/pixi.lock` 管理；子模块配置保持可组合性
- 使用 `_<task_name>` 下划线格式来隐藏不暴露给人类的任务，逻辑上解耦，使用上一体
- 所有构建与任务通过 Pixi 运行，例如 `pixi run pnpm build`，而不是 `pnpm run pixi test`
- pnpm 仅管理 JavaScript 项目自身依赖与js脚本，不负责 Android 项目任务和编译任务
- 优先使用 Conda 包；允许多个模块使用相同依赖，并保持版本同步
- 优先使用共享 solve group，不重复独立求解环境
- 使用 `depends-on` 组合任务，避免在命令字符串中串联任务
- 编译任务应声明 input、output 和 force 行为，以获得正确缓存
- 统一任务使用相对 `ci/pixi.toml` 的 `cwd`；子模块任务使用相对其所属 `pixi.toml` 的 `cwd`，不使用绝对路径
- 移动或创建 `pixi.toml` 时，必须同步检查任务 `cwd` 与 Workspace 引用
- `ci/pixi.lock`、子模块锁文件和 `pnpm-lock.yaml` 必须提交至 Git ！！！
- 构建配置和资源策略记录在 `ci/config/`，不保存依赖用户本机设备环境的配置

#### 依赖维护与可复现发布

- Pixi 可执行程序由调用方安装，当前建议使用 `0.72.2` 或更高版本；仓库不锁定用户机器上的 Pixi 本体版本
- pnpm 当前建议使用 `11.13` 或更高版本，并应随受管理 package 的依赖要求与安全支持状态持续更新
- 全仓 dependency registry 必须逐项登记生态、manifest、lockfile 或求解基线、所有者、更新命令和管理状态；Git submodule、vendored 源码、模板与独立工程不得由目录扫描直接决定或改写
- 人应当在准备依赖变更、完整验证或发布前主动执行统一的 `dependency_update_check` task，取得全仓依赖诊断报告
- `dependency_update_check` 必须覆盖 Python package、Conda/Pixi environment、pnpm workspace 的直接与传递依赖，并纳入 Gradle 依赖和外部制品能够取得的更新信息
- 诊断严格检查已公布安全漏洞，报告 advisory ID、数据源、严重性、受影响版本范围、依赖路径和已知修复版本；同时报告普通可用更新、版本约束、锁文件状态和源码适配需求
- 更新检查只产生结构化报告，不在检查过程中修改 manifest、lockfile 或源码；报告记录检查时间、数据源、依赖输入 hash、当前版本、候选版本、安全通告和审查结论
- 版本化 security policy 必须按生态指定 advisory 数据源、别名去重、受影响范围匹配和报告有效期；任一必需数据源不可用时报告为不完整，不能满足 release 门禁
- 所有公开构建命令在动作摘要中显示依赖检查状态、可更新依赖数量、安全漏洞数量和 `dependency_update_check` 的执行提示
- 普通 build 与 QA 不强制联网诊断，但必须展示报告状态和警告；release 准备必须具有与当前依赖输入 hash 匹配且仍在规定有效期内的报告
- 人未提前生成可用报告时，release 准备阶段显式执行一次完整 `dependency_check`
- 已公布安全漏洞是 release 阻塞项；不涉及安全问题的版本更新只产生警告并进入依赖维护报告
- 发现必要更新时，应在独立修改中更新依赖声明与锁文件；外部 API 或行为变化需要同步修改项目代码、文档和验证，不以现有代码尚未适配为理由长期停留在存在已知安全问题的版本
- 依赖更新前后尽可能执行同一组静态检查、测试、构建、APK 审计和资产审计，并生成结构化对比报告；确认差异符合预期后，统一更新全仓库依赖声明、lockfile、外部制品记录和依赖求解器基线
- 暂时不能更新的依赖必须记录阻塞原因、影响、负责人和处理期限；安全支持结束、存在已知可利用问题或超过项目维护期限的依赖不得进入发布输入
- release profile 只使用已经审查并提交的 manifest 与 lockfile，依赖安装使用冻结模式，构建过程中不解析新的依赖版本
- 发布输入记录 `ci/pixi.lock`、`pnpm-lock.yaml`、Gradle 依赖声明和外部制品 manifest 的 hash，并必须关联与这些 hash 匹配的依赖诊断报告
- 锁文件用于复现某次已审查的构建输入，不代表停止依赖维护；依赖更新检查与冻结构建是两个明确阶段

依赖更新由人负责测试判断，脚本负责重复且可审计的准备工作：

- `dependency_update prepare` 检查干净的依赖维护专用 worktree、记录旧依赖图与输入 hash，并构建升级前测试包；人使用该测试包执行升级前测试并记录结果
- `dependency_update apply` 串行更新能够自动处理的依赖声明、lockfile 和求解器基线，不并行修改多个依赖生态
- 依赖更新可能改变命令自身使用的 Python、Conda、Pixi 或 pnpm 环境；到达这种边界时，命令写入明确检查点，输出需要人执行的 bootstrap 或手动升级命令，以及唯一的 `dependency_update resume` 启动命令
- `dependency_update resume` 从检查点继续未完成阶段，不重复已经确认的写入，并构建升级后测试包、依赖图和结构化差异报告
- 人使用升级前后测试包执行同一测试集合，负责确认功能、性能、数据和用户空间差异；脚本不代替人的测试结论
- `dependency_update finalize` 必须在新依赖输入 hash 上重新执行完整安全诊断，校验前后人工测试记录、允许 diff 范围、全部 registry 条目、manifest、lockfile、外部制品记录和求解基线，并输出拟提交文件清单
- 同一工作区同一时间只允许一个依赖更新会话；命令使用会话锁和状态文件拒绝并行更新
- 依赖更新必须形成专用 commit，只包含依赖声明、lockfile、求解器基线、必要的兼容代码、对应测试和文档；严禁混入无关功能、重构或格式化
- 依赖更新 commit 可以处于正常发布提交序列中，不要求创建独立版本或独立发布节点

#### Android 工具链与本地状态

- Pixi 提供 OpenJDK 与 `android-tools`；`android-tools` 只提供 ADB 与 fastboot
- Android command-line tools 由工具链清单固定版本与下载 hash，环境准备任务使用 `sdkmanager --sdk_root` 安装 Gradle 声明的 Android 组件
- Gradle 声明并消费 `compileSdk`、`buildToolsVersion`、`ndkVersion` 与 CMake `version`，不承担 SDK 组件安装
- Android Clang、LLD、sysroot 与交叉编译器随 NDK 版本整体确定
- 默认 `ANDROID_HOME` 为仓库根 `.local/android-sdk/`，默认 `GRADLE_USER_HOME` 为仓库根 `.local/gradle/`；构建入口提供明确的目录重定向参数
- `JAVA_HOME`、`ANDROID_HOME` 与 `GRADLE_USER_HOME` 只作用于 Pixi task 及其子进程，不写入 shell profile、注册表或系统环境变量
- 下载后的 SDK、NDK、CMake 与 Gradle 状态属于本地可重建状态，不放入 `ci/tools/`

### Android 编译器与资源裁剪契约

- `qaFullDebug` 与 `qaLiteDebug` 使用 D8，并保持 `isMinifyEnabled=false`
- `qaFullRelease` 与 `prodFullRelease` 使用 R8，并保持 `isMinifyEnabled=true`
- 编译器选择由 build type 固定，不提供运行时切换参数
- 资源裁剪独立于 R8 配置；修改 release 资源裁剪设置时，必须同步完成 keep rules 验证与 APK 内容审计

### 构建缓存

- 建议将中间缓存置于 SSD，并为可信缓存目录配置杀毒软件排除项
